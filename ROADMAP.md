# Roadmap

Активные задачи расположены по убыванию риска и с учетом зависимостей: сначала предотвращение потери данных, затем надежность операций, архитектурные enabler'ы, UX и сопровождаемость. Обязательный enabler всегда расположен выше зависимой задачи; если нужен только фрагмент нижней задачи, он выделяется и поднимается отдельно. Активный граф зависимостей не должен содержать циклов. Независимый аудит 2026-07-14 переоткрыл P0: обычные исключения, rollback и Windows safety-check покрыты хорошо, но apply пока не защищен от stale snapshot, аварийного завершения процесса, hard link на lock-файл и части XML round-trip рисков. Новый функционал не должен задерживать исправления P0–P1.

Добавление, декомпозиция и приёмка задач выполняются по [правилам планирования roadmap](docs/roadmap-workflow.md). Отметка `[x]` означает, что задача прошла согласованные проверки и итоговое acceptance review; наличие основной реализации само по себе не считается завершением. Версии поднимаются только после соответствующей вехи: `1.0-alpha` публикуется как `1.0.0-alpha.N`, `1.0-beta` — как `1.0.0-beta.N`, стабильный выпуск — как `1.0.0`; nightly не заменяет release gate.

## P0 — целостность данных и безопасный apply

- [x] **Зафиксировать непроверенные fail-closed инварианты mutation core перед P0-рефакторингом.** Зачем: следующие P0 меняют общий XML/file I/O, lock и transaction state, а часть существующих отказов trust/configuration/cleanup не закреплена тестами и может регрессировать при зеленом suite. Приёмка: regression-тесты покрывают duplicate/missing destination, половинчатую trust-конфигурацию, разные anchors, отмену пустой транзакции, поврежденный и типизированно неверный lock owner, `lstat`/`resolve` failures и canonical escape; для каждого случая проверены outcome, file state/error, ноль commit и внешних записей, отсутствие temp; измерение этих ветвей общим branch-coverage gate остается в P2 и не блокирует поведенческую приёмку P0.
- [x] **Зафиксировать и обеспечить безопасный XML round-trip.** Зачем: стандартный `ElementTree` parser уже воспроизводимо удаляет comments и processing instructions, поэтому apply меняет больше разрешенных узлов, а зависимая canonicalization не может корректно определить mixed/nested семантику до фиксации codec-контракта. Приёмка: shared XML I/O владеет semantic/lexical контрактом для unknown elements/attributes, comments, PI, namespaces/prefixes, declaration, BOM, Unicode и EOL; parser сохраняет обязательные узлы; end-to-end fixtures проходят `read → one-target mutation → write → reread` для dedupe и replace-paths и доказывают неизменность всего остального; большие/namespaced XML и cancellation остаются покрыты. Выполнить до nested/mixed dedupe, snapshot guard, memory/codec refactor и любых новых XML-мутаций.
- [x] **Закрыть оставшийся gap консервативной дедупликации вложенного и mixed XML поверх готового round-trip-контракта.** Зачем: задача повторно открыта после adversarial review исправления `4d493f7`: глобальный signature fallback смешивает одинаковые lexical QName из разных inherited namespace scopes, Python `str.strip()` ошибочно считает NBSP форматированием, а cross-key duplicate может скрыть существующую ambiguity; независимое review `6c7adcb` дополнительно выявило зависимость `#parent-content` от позиции representative и удаление одинаковой записи из другого parent context. Приёмка: порядок игнорируется только для непосредственных полей `AdditionalApplication`; logical group является связной компонентой по совпадающему primary key либо полному мультимножеству нормализованных значений всех `GameID` / `ApplicationPath`, а automatic duplicate требует равенства полного namespace-aware signature под одним непосредственным XML-parent внутри этой компоненты; разные parents дают `#parent` ambiguity; canonical identity учитывает lexical QName и namespace URI, включая унаследованный scope; formatting-only означает только XML-пробелы `#x20/#x9/#xD/#xA`; significant element/parent content и все неоднозначные варианты сохраняются независимо от first/kept/duplicate роли; adversarial-тесты воспроизводят дефекты на соответствующих baseline и проходят на кандидате; итоговую отметку `[x]` ставит независимый проверяющий после повторного acceptance gate.
- [ ] **Защитить mutation lock и доверенные mutable-файлы от hard link и handle-race.** Зачем: существующий `.launchbox-utils-mutation.lock`, являющийся hard link на внешний файл, проходит path guard и при записи metadata портит внешний target; последующие snapshot и journal не должны создавать ещё один path-based I/O контракт. Приёмка: общий Windows handle-based примитив после открытия проверяет file identity, link count, regular-file и reparse semantics и доступен последующим snapshot/journal слоям; до успешной проверки lock не обрезается; чтение owner metadata ограничено по размеру; hard-link/swap сценарии завершаются fail-closed без изменения sentinel; обязательные real-Windows race-тесты сохраняют обычную конкуренцию двух процессов и гарантируют завершение обоих процессов.
- [ ] **Связать commit с точным XML snapshot, из которого построен plan.** Зачем: внешний редактор может изменить и закрыть XML после чтения; текущий backup сохранит новую версию, а commit молча затрет ее старым деревом. Приёмка: repository из одного проверенного read handle создает immutable `SourceSnapshot` / `SourceFingerprint` с file identity и SHA-256 тех же байтов, которые распарсены; этот низкоуровневый контракт принадлежит repository/transaction layer и не зависит от будущего P1 operation controller; `XmlMutation` несет expected fingerprint; все источники сверяются до первого commit и повторно непосредственно перед границей commit; mismatch дает структурированный `source_changed`, для `replace-paths` — ноль commit, достоверные states и manifest; regression-тесты меняют файл после parse и после stage для обеих мутаций.
- [ ] **Сделать многофайловый apply crash-consistent и обеспечить низкоуровневый recovery.** Зачем: `os._exit`, kill или сбой питания после первого `os.replace` оставляет частично примененный `replace-paths`, backups без итогового `manifest.json` и не запускает Python rollback. Приёмка: до первого replace атомарно и устойчиво записан `in_progress` journal с destinations, source/new hashes и проверенными backups; P0 transaction layer владеет форматом journal, его безопасным чтением, state machine, обнаружением незавершенного run, блокировкой нового apply и идемпотентным recovery API/command; прогресс commit фиксируется так, чтобы следующий запуск однозначно определял состояние; process-crash тесты покрывают точки до/после каждого replace и обновления journal; после recovery все XML либо старые, либо согласованно новые. Retention, guided workflow, cleanup policy и GUI остаются в P1 и ниже.
- [x] **Реализовать базовую консервативную дедупликацию `AdditionalApplication`.** Автоматически удаляются канонически одинаковые варианты верхнеуровневых полей; различия в `CommandLine`, `AutoRunBefore` / `AutoRunAfter`, настройках эмулятора и других известных полях остаются ambiguous. Оставшийся nested-order gap отслеживается отдельной P0-задачей выше.
- [x] **Ввести транзакционный mutation flow `plan → validate all → backup all → stage → commit → rollback` для контролируемых ошибок процесса.** Для связанного `replace-paths` уже выполняется rollback ранее замененных файлов при пойманной ошибке; для независимых dedupe-файлов есть явный partial outcome. Crash-consistency вынесена в отдельную P0-задачу выше.
- [x] **Сделать результаты мутаций достоверными.** Использовать состояния `planned` / `prepared` / `committed` / `failed` / `rolled_back` как единственный источник истины. GUI, CLI, отчеты и итоговый apply manifest одинаково различают success, partial, failed и rolled back.
- [x] **Защитить мутации от штатных повторных и параллельных запусков.** Межпроцессный lock, UUID операции, уникальные каталоги backup и временные файлы, manifest с исходными путями и хешами реализованы; существующий backup не перезаписывается. Hostile hard-link случай вынесен в отдельную P0-задачу выше.
- [x] **Исправить GUI lifecycle во время операций.** Использовать non-daemon worker, обрабатывать `WM_DELETE_WINDOW`, не разрешать завершение процесса в commit/rollback-фазе и разрешать отмену только до commit. Проверить закрытие окна на стадиях scan, stage, commit и rollback.
- [x] **Ограничить платформенные XML доверенным каталогом.** Имя платформы является одним допустимым компонентом; canonical path остается непосредственным потомком `Data/Platforms`; traversal, absolute/UNC paths, DOS reserved names, symlink/reparse points и junctions отклоняются. Hard links и полностью handle-relative I/O остаются отдельным P0 hardening.
- [x] **Исправить Windows safety-check и сделать его fail-closed.** Объявить корректные WinAPI-сигнатуры, использовать `WinDLL(..., use_last_error=True)`, структурированно определять `LaunchBox.exe` и `BigBox.exe` без зависимости от локали `tasklist`, добавить timeout и повторную проверку непосредственно перед commit.
- [x] **Добавить реальные Windows integration-тесты safety-check.** Проверить эксклюзивный handle из отдельного процесса, запущенные LaunchBox и Big Box, access denied, локализованный вывод, ошибку/timeout диагностики и гонку между предварительной проверкой и commit.

### Веха A — повторный аудит безопасного apply и `1.0-alpha`

- [ ] **Провести независимый аудит после закрытия P0.** Зачем: реализация каждого отдельного исправления не доказывает совместную сохранность данных на границах parse/plan/stage/commit/recovery. Приёмка: все P0 отмечены `[x]`; проверяющий не является автором проверяемых изменений; аудит заново охватывает безопасность, корректность, архитектуру, производительность, сопровождаемость и осмысленность тестов, включая реальные Windows, hard-link/swap и process-crash сценарии; каждое замечание воспроизведено и добавлено в roadmap выше зависимых задач; открытых P0, Blocker и Regression не осталось.
- [ ] **Зафиксировать distribution/config contract и минимальный smoke для alpha.** Зачем: console script уже публикуется для wheel, но default INI вычисляется рядом с package/site-packages и может быть read-only; без принятого контракта alpha release и последующий E2E gate не знают, какие поверхности обязательны. Приёмка: либо pip/wheel официально поддержан с user-writable config location, либо surface явно снят из metadata/docs; EXE config discovery и содержимое архива определены отдельно; для каждой поддерживаемой alpha surface зафиксирован минимальный smoke на clean environment, включая CLI `--help`/dry-run и GUI startup-close там, где применимо.
- [ ] **Выпустить `1.0.0-alpha.1` после успешной вехи A.** Зачем: alpha должна обозначать первую внешне проверенную сборку с безопасным базовым mutation flow, а не просто текущий nightly. Приёмка: аудит вехи A и distribution/config contract приняты; версия синхронна в `pyproject.toml`, `launchbox_tools.__version__`, CLI/GUI и metadata артефакта; tag и release создаются из чистого commit, помечены prerelease и содержат release notes с известными ограничениями; все поддерживаемые surfaces проходят минимальный smoke из принятого контракта; опубликованы проверяемые SHA-256.

## P1 — надежность операций и восстановление

- [ ] **Замкнуть output boundary и доказуемое владение отчетами.** Зачем: cleanup следует заранее созданному junction каталога платформы и может удалить одноименный файл вне `output_dir`; legacy cleanup также удаляет файлы только по имени без marker владения. Приёмка: canonical output anchor запрещает symlink/junction/reparse во всех дочерних путях; summary/detail hard link не изменяет внешний target; stale удаляется только по run manifest/marker; предпочтительно отчеты публикуются отдельным run-directory; существующие stale-cleanup тесты, закрепляющие удаление немаркированных файлов, удалены и заменены ownership/fail-closed тестами.
- [ ] **Поднять единый operation controller и immutable plan в общий application layer поверх P0-контрактов.** Зачем: CLI и GUI дублируют orchestration, summary и report flow; без общего plan невозможно надежно реализовать preview, partial audit и единые diagnostics. Приёмка: один `RunContext`/plan DTO и `OperationSummary` обслуживают CLI и GUI; adapters только валидируют ввод и отображают результат; plan включает готовые P0 `SourceSnapshot` / `SourceFingerprint`, но не переопределяет их семантику или recovery state machine, и содержит canonical inputs, affected XML и фактические output/backup paths; публичные DTO не удерживают `ElementTree` и не используют скрытую `_state_source`, а `vars`/`asdict`/copy согласованы; outcome, progress, errors, reports и manifest одинаковы на обеих поверхностях.
- [ ] **Возвращать явную ошибку для неизвестного `--platform`.** Зачем: опечатка сейчас закреплена тестом как успешный пустой результат. Приёмка: dedupe и replace-paths до backup/manifest возвращают stable `unknown_platform`; CLI exit 1, GUI локализует ошибку и показывает доступные имена; case-insensitive точное совпадение работает; `test_dedupe_additional_apps_can_filter_platform` с неверным ожиданием удален и заменен operation/CLI/GUI matrix.
- [ ] **Исключить ложные path-replacement изменения и no-op apply.** Зачем: canonical-equivalent `--old` и `--new` сейчас создают replacements и переписывают XML без полезного результата, усиливая round-trip риск. Приёмка: эквивалентные roots отклоняются до scan либо дают ноль mutations; отдельная запись не планируется, если итоговый text эквивалентен исходному; проверены case, slash style, trailing separator и absolute/relative формы; no-op не создает backup/commit, пустой manifest допустим только по явно принятому контракту.
- [ ] **Заменить общее подтверждение apply на preview того же immutable plan.** Зачем: пользователь сейчас подтверждает абстрактное сообщение, не видя scope изменения. Приёмка: preview показывает количество и типы изменений, список XML, ключевые параметры, риски и фактический каталог backup; безопасный ответ по умолчанию — «Нет»; apply использует тот же plan и P0 fingerprint guard, а не скрытый повторный scan; CLI имеет машинно-читаемый эквивалент preview.
- [ ] **Продолжать audit после локальной ошибки и честно обозначать неполный scan.** Зачем: parse/OSError одной платформы обрывает run, а `rglob` OSError превращается в warning при общем exit 0, поэтому частичный результат выглядит полным. Приёмка: общий audit result и каждая платформа имеют `success/partial/failed/cancelled`; unreadable subtree, missing/corrupt database дают partial и non-zero CLI status, но результаты исправных платформ и reports сохраняются; unsafe catalog/name остается fail-closed для всего run; CLI, GUI, summary и details показывают одно состояние.
- [ ] **Сделать публикацию отчетов атомарной и безопасной для CSV/Windows имен.** Зачем: writer failure может обрезать прежний отчет, CSV принимает Excel formulas, а sanitized имена не учитывают DOS devices и устойчивые коллизии. Приёмка: каждый CSV/TXT или целый run staged и публикуется atomic replace только внутри защищенного output boundary; при injected failure старый отчет цел и temp удален; единый sanitizer нейтрализует leading whitespace/control chars и `=`, `+`, `-`, `@`; `CON`/`PRN`/`AUX`/`NUL`/`COM1`, две одинаково sanitized платформы и stale mapping покрыты тестами; CSV проверяется обратным parsing по полям.
- [ ] **Сделать INI literal, атомарным и диагностируемым.** Зачем: стандартная интерполяция ломает путь вроде `100% Fun`, а autosave пишет файл напрямую и не показывает ожидаемые preflight/config ошибки. Приёмка: везде используется `ConfigParser(interpolation=None)`; save выполняется через exclusive temp + atomic replace и сохраняет старый INI при сбое; GUI локализует autosave, missing/corrupt/unreadable XML до confirmation, CLI ловит `configparser.Error` без traceback; `%` round-trip и injected-failure тесты обязательны.
- [ ] **Определить эксплуатационную политику backup поверх готового P0 journal/recovery.** Зачем: даже корректные backups и низкоуровневый recovery бесполезны без понятного retention и операторского процесса. Приёмка: policy использует P0 journal reader и recovery API без повторной интерпретации state machine; документированы сроки/лимиты хранения, проверка hash/readability, guided rollback/restore и безопасная очистка только принадлежащих утилите каталогов; есть dry-run cleanup и end-to-end проверка эксплуатационного recovery flow; задача не владеет обнаружением незавершенного run или выполнением низкоуровневого recovery и предшествует GUI выборочного восстановления.

## P2 — UX, архитектура и качество

- [ ] **Ввести branch coverage и обязательные Windows/Tk quality gates.** Зачем: 221 тест проходят быстро, но ориентировочное line coverage около 80%, branch coverage не измеряется, а security/real-Tk тесты могут тихо skip. Приёмка: CI запускает `coverage run --branch`, фиксирует baseline без регрессии и отдельный gate для critical safety modules; Windows integration job падает при skip Tk/symlink/junction/process scenarios и имеет timeout; project warnings собираются явно, deprecated `locale.getdefaultlocale` удален; добавлены `ruff` и поэтапный `mypy` без массового подавления.
- [ ] **Добавить end-to-end smoke поддерживаемых пользовательских entrypoints.** Зачем: внутренние функции покрыты лучше, чем реальные поверхности, зафиксированные distribution contract. Приёмка: subprocess-smoke проверяет audit fixture, stdout/stderr, exit codes и реальные reports для source и каждой поддерживаемой installed surface; если wheel/console script поддержан, он проверяется в clean venv; GUI создается, запускает реализованные actions и закрывается без полного mainloop; nightly после build запускает CLI EXE `--help`/dry-run и GUI startup-close smoke.
- [ ] **Удалить или переписать оставшиеся тесты с низкой и ложноположительной ценностью.** Зачем: отдельные тесты дублируют строки parser, а некоторые проверки закрепляют неверные контракты; P0 fixture/race enablers уже входят в соответствующие P0-задачи. Приёмка: два `test_cli_parser_supports_*` заменены table-driven parser/dispatch matrix; unsafe stale-cleanup и unknown-platform ожидания заменены защитными сценариями при реализации соответствующих P1-контрактов; RU/EN проверяются по равенству полного набора ключей. Real-Windows, rollback и dry-run/apply пары ради процента не удаляются.
- [ ] **Ввести явное busy-состояние до любого I/O.** Зачем: повторный click сейчас выполняет catalog parse, `tasklist` и confirmation на Tk thread до поздней проверки `is_busy()`. Приёмка: controller атомарно фиксирует busy и snapshot параметров до preflight; конфликтующие поля, navigation и actions блокируются; второй запуск не делает safety-check/confirmation; progress, duration и итог success/partial/failure отображаются; ожидаемые preflight errors локализованы.
- [ ] **Создать воспроизводимый performance baseline до оптимизаций.** Зачем: текущие решения не имеют измеренных time/RSS/I/O budgets. Приёмка: benchmark harness покрывает 10k/100k/500k файлов, одинаковые/overlapping ROM folders, большие XML, dry-run/apply/cancel/rollback; сохраняет wall time, peak RSS, число filesystem probes и process launches; пороги сравниваются с зафиксированным baseline на документированной Windows-машине.
- [ ] **Зафиксировать политику поддерживаемых Python версий без преждевременного drop.** Зачем: metadata/CI сейчас заявляют 3.10–3.13, локальный suite проходит на 3.14, а технического основания немедленно удалять 3.10/3.11 не зафиксировано. Приёмка: принят EOL/support policy; `requires-python`, classifiers, README и CI совпадают; 3.14 добавлен в matrix; решение о drop 3.10/3.11 принимается отдельно по policy; min/max versions проходят полный discovery, warning gate и nightly build.
- [ ] **Усилить supply-chain и выпуск EXE.** Зачем: build dependency и Actions используют плавающие версии, artifact не имеет проверяемой цепочки происхождения. ### Функциональный backlog

Функции ниже не входят автоматически в release scope `1.0.0` и рассматриваются после P0–P1. Backlog не является готовой последовательностью реализации: перед продвижением задачи в активный приоритет для неё строится ациклический граф зависимостей, а минимальные enablers поднимаются раньше зависимого результата. Любая новая мутация обязана использовать общий plan/transaction/controller flow, snapshot guard и crash-recovery journal; включение функции в alpha/beta/1.0 оформляется отдельным решением со своими критериями приёмки.
