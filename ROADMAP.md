# Roadmap

Активные задачи расположены по убыванию риска: сначала предотвращение потери данных, затем надежность операций, UX и сопровождаемость. До выполнения P0 apply-операции нельзя считать полностью безопасными для рабочей базы LaunchBox. Новый функционал не должен задерживать исправления P0–P1.

## P0 — целостность данных и безопасный apply

- [x] **Сделать дедупликацию `AdditionalApplication` консервативной.** Автоматически удалять только канонически одинаковые записи. Совпадения по `GameID` и пути с различиями в `CommandLine`, `AutoRunBefore` / `AutoRunAfter`, настройках эмулятора или других значимых полях считать ambiguous и оставлять для ручного решения. Закреплено fixture-тестами.
- [x] **Ввести транзакционный mutation flow `plan → validate all → backup all → stage → commit → rollback`.** Для связанного `replace-paths` обеспечить all-or-nothing; для независимых dedupe-файлов — явный partial outcome. Ошибка записи любого XML должна приводить к rollback уже замененных файлов и не оставлять ложных `applied`.
- [x] **Сделать результаты мутаций достоверными.** Использовать состояния `planned` / `prepared` / `committed` / `failed` / `rolled_back` как единственный источник истины. GUI, CLI, отчеты и итоговый apply manifest одинаково различают success, partial, failed и rolled back.
- [x] **Защитить мутации от повторных и параллельных запусков.** Добавить межпроцессный lock, UUID операции, уникальные каталоги backup и временные файлы, manifest с исходными путями и хешами. Никогда не перезаписывать существующий backup; очищать временные файлы в `finally`. Покрыть параллельный apply и два запуска в одну секунду.
- [ ] **Исправить GUI lifecycle во время операций.** Использовать non-daemon worker, обрабатывать `WM_DELETE_WINDOW`, не разрешать завершение процесса в commit/rollback-фазе и разрешать отмену только до commit. Проверить закрытие окна на стадиях scan, stage, commit и rollback.
- [ ] **Ограничить платформенные XML доверенным каталогом.** Имя платформы должно быть одним допустимым компонентом; canonical path обязан оставаться непосредственным потомком `Data/Platforms`. Для мутационных целей отклонять traversal, absolute/UNC paths, DOS reserved names, reparse points и junctions.
- [ ] **Исправить Windows safety-check и сделать его fail-closed.** Объявить корректные WinAPI-сигнатуры, использовать `WinDLL(..., use_last_error=True)`, структурированно определять `LaunchBox.exe` и `LaunchBox Big Box.exe` без зависимости от локали `tasklist`, добавить timeout и повторную проверку непосредственно перед commit.
- [ ] **Добавить реальные Windows integration-тесты safety-check.** Проверить эксклюзивный handle из отдельного процесса, запущенные LaunchBox и Big Box, access denied, локализованный вывод, ошибку/timeout диагностики и гонку между предварительной проверкой и commit.

## P1 — надежность операций и восстановление

- [ ] **Заменить общее подтверждение apply на проверяемый preview.** Показывать число изменений, список XML, ключевые параметры и каталог backup; безопасный ответ по умолчанию — «Нет». Перед commit проверять, что просмотренный snapshot файлов не изменился.
- [ ] **Зафиксировать контракт XML round-trip.** Сохранять неизвестные элементы и атрибуты; определить требования к comments, processing instructions, namespace prefixes, declaration, BOM и переносам строк. После записи выполнять семантическую проверку и добавить регрессии для namespaces, comments, PI, Unicode и больших XML.
- [ ] **Сделать запись INI и отчетов атомарной.** Отключить интерполяцию `ConfigParser`, использовать stage + atomic replace, корректно показывать ошибки autosave. Для CSV нейтрализовать formula injection с префиксами `=`, `+`, `-`, `@`; защитить имена отчетов от `CON`, `PRN`, `AUX`, `NUL`, `COM1` и коллизий.
- [ ] **Разделять сбой операции и сбой отчета.** Успешная XML-мутация не должна выглядеть неуспешной только из-за writer error, а ошибка отчета не должна скрывать фактический commit. Manifest должен позволять восстановить итог независимо от отчета.
- [ ] **Продолжать audit после ошибки отдельной платформы.** Возвращать структурированный результат с ошибкой, сохранять результаты исправных платформ и отражать partial outcome в CLI, GUI, summary и detail reports.
- [ ] **Определить политику backup.** Документировать хранение, проверку, восстановление и очистку; удалять только backup и stale-файлы, однозначно принадлежащие утилите. Добавить проверку читаемости backup и пошаговый recovery workflow.

## P2 — UX, архитектура и качество

- [ ] **Добавить единый operation controller для CLI и GUI.** Использовать immutable run context и общий `OperationSummary`; убрать дублирование orchestration и подсчета итогов. Контроллер должен управлять фазами, progress, отменой read-only/pre-commit стадий и фактическим каталогом отчетов запуска.
- [ ] **Ввести явное busy-состояние формы.** Во время запуска блокировать конфликтующие поля, навигацию и action-кнопки, показывать snapshot параметров, прогресс, длительность и итог success/partial/failure. Не выполнять safety-check и confirmation повторного apply после того, как уже запущена другая операция.
- [ ] **Улучшить доступность GUI.** Устранить накопление объектов `Tooltip`, добавить keyboard focus и accelerators, понятные локализованные названия вместо зависимых от tooltip кнопок `...` и `>`, корректный возврат фокуса после ошибки и responsive layout для RU/EN при 100–200% DPI.
- [ ] **Декомпозировать `launchbox_tools/gui/app.py` после стабилизации controller.** Разделить каркас окна, панели операций, execution state и форматирование результата. Отделить доменные модели от `ElementTree`-узлов и зафиксировать целевую структуру в `ARCHITECTURE.md`.
- [ ] **Оптимизировать только после benchmark.** Измерить библиотеки на 10k/100k/500k файлов, overlapping ROM folders и большие XML; затем кешировать одинаковые сканы, применять `scandir`, избегать повторных `resolve`/`exists`, не обходить junctions и не удерживать одновременно все XML-деревья.
- [ ] **Разделить тесты по подсистемам и усилить quality gates.** Выделить config, XML, safety, operations, reports, CLI и GUI; сделать GUI smoke обязательным, добавить `ruff`, постепенный `mypy`, Python 3.14 в CI и заменить deprecated `locale.getdefaultlocale` до Python 3.15.
- [ ] **Усилить сборку и выпуск EXE.** Зафиксировать build-зависимости и GitHub Actions по commit SHA, минимизировать permissions, публиковать SHA-256, SBOM и provenance; предусмотреть Authenticode-подпись.

## P3 — сопровождаемость и документация

- [ ] Возвращать явную ошибку для неизвестного `--platform`, удалить неиспользуемый код и унифицировать Windows-safe naming/cleanup между writer-модулями.
- [ ] Синхронизировать README и `ARCHITECTURE.md` с operation controller, состояниями операций, preview/apply, фактической навигацией GUI, восстановлением backup и политикой UTF-8.
- [ ] Унифицировать пользовательские термины RU/EN: dry-run/apply, отчет/лог, success/partial/failure/rollback; не показывать «Готово» после частичных ошибок.

### Функциональный backlog

Функции ниже рассматриваются после P0–P1. Любая новая мутация обязана использовать общий transaction/controller flow.

- [ ] **Аудит внутренней целостности и правил качества базы.** Находить пустые/повторяющиеся Game ID, orphaned `AdditionalApplication` и `PlaylistGame`, отсутствующие Emulator ID и несогласованные платформы. Затем добавить проверяемые профили для обязательных полей, путей, регионов, версий и пользовательских полей. V1 строго read-only.
- [ ] **Массовый аудит готовности и диагностика запуска игр.** Проверять цепочку ROM → EmulatorId → executable → associated/default platform → RetroArch core → BIOS/dependencies и показывать вычисленную команду запуска с рабочим каталогом, параметрами и дополнительными приложениями. V1 ничего не запускает и строит сводку причин ошибок.
- [ ] **Верификация ROM-наборов по DAT-каталогам.** Импортировать локальные DAT No-Intro, Redump и MAME; сверять обычные файлы и содержимое ZIP по CRC32/SHA-1/SHA-256 и показывать корректные, поврежденные, лишние и отсутствующие ROM. V1 read-only, один DAT на платформу.
- [ ] **Поиск перемещенных или переименованных ROM по хешу.** Сопоставлять `missing_on_disk` и `not_in_database` сначала по размеру, затем SHA-256; apply разрешать только для уникального соответствия 1:1. Перепакованные архивы V1 не считать эквивалентными.
- [ ] **Анализ дублирующегося дискового пространства ROM.** Искать идентичные ROM, CHD, ISO и архивы по размеру и хешу, показывать использующие их записи LaunchBox и потенциально освобождаемое место. V1 только формирует отчет; hardlink-дедупликацию рассматривать отдельно после проверки ограничений NTFS и rollback.
- [ ] **Просмотр, сравнение и выборочное восстановление backup.** Сравнивать backup утилиты и LaunchBox с текущим `Data`; V1 восстанавливает только целый выбранный XML после сохранения текущей версии, без element-level merge.
- [ ] **Восстановление основного файла из `AdditionalApplication`.** V1: dry-run с однозначным кандидатом, явный preview и apply через общий mutation flow.
- [ ] **Аудит переносимости сборки.** Формировать read-only manifest абсолютных, внешних и отсутствующих путей для Games, Additional Apps, эмуляторов, конфигурации и явно описанных dependencies.
- [ ] **Сравнение и объединение двух установок LaunchBox.** Сопоставлять платформы, игры, дополнительные приложения и эмуляторы по Database ID, хешам и составным ключам. V1 — read-only отчет о различиях и конфликтах; выборочный импорт добавлять только через общий mutation flow.
- [ ] **Экспорт избранного для портативной сборки.** Поддержать режимы «только основной файл» и «все связанные файлы», collision policy и manifest результата.
- [ ] **Поддержка многодисковых игр.** Сначала определить модель связей и read-only аудит, затем безопасные операции.
