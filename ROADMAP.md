# Roadmap

Активные задачи расположены по убыванию важности: от рисков потери данных к надежности, UX, сопровождаемости и новым функциям. До выполнения P0 apply-операции не следует считать безопасными для реальной базы LaunchBox.

## P0 — блокирующие целостность данных

- [ ] Исправить Windows safety-check: объявить корректные WinAPI-сигнатуры и использовать `use_last_error`, определять `LaunchBox.exe` и `BigBox.exe` по структурированным данным, добавить timeout и блокировать apply при любой ошибке диагностики (`fail-closed`).
- [ ] Добавить реальные Windows integration-тесты safety-check: эксклюзивный handle из отдельного процесса, запущенный Big Box, ошибка и timeout определения процессов, а также повторная проверка непосредственно перед commit.

## P1 — высокий приоритет

- [ ] Сделать дедупликацию `AdditionalApplication` консервативной: автоматически удалять только канонически одинаковые записи; совпадения только по `GameID` и пути с любыми другими различиями считать ambiguous и оставлять для ручного решения. Закрепить сценарии с разными `CommandLine`, `AutoRunBefore` / `AutoRunAfter` и настройками эмулятора.
- [ ] Валидировать пути к платформенным XML: имя платформы должно быть одним допустимым компонентом, итоговый путь обязан оставаться внутри `Data/Platforms`, а reparse points и junctions для мутационных целей должны отклоняться.
- [ ] Ввести транзакционный mutation flow `plan → validate all → backup all → stage → commit → rollback`: для связанного `replace-paths` обеспечить all-or-nothing, для независимых dedupe-файлов сохранять явный partial outcome. Тест с ошибкой записи второго XML должен подтверждать rollback и отсутствие ложного `applied`.
- [ ] Защитить мутации от параллельных запусков: добавить межпроцессный lock, UUID операции, уникальные временные файлы, backup с сохранением относительной структуры и manifest с путями и хешами; очищать временные файлы в `finally`. Покрыть параллельные apply и два запуска в одну секунду.
- [ ] Исправить GUI lifecycle: использовать non-daemon worker, обрабатывать `WM_DELETE_WINDOW`, запрещать аварийный выход в commit-фазе, сохранять отложенные настройки и разрешать отмену только до commit. Добавить тест закрытия окна во время apply.
- [ ] Заменить общее подтверждение apply на preview с числом изменений, списком затрагиваемых XML и путем к backup; безопасный ответ по умолчанию — «Нет», а commit должен проверять неизменность просмотренного snapshot.
- [ ] Ввести терминальные состояния `success` / `partial` / `failed` / `rolled_back`: не показывать «Готово» после частичных ошибок, раздельно сообщать сбой мутации и сбой записи отчета, сохранять правдивый итог в GUI, CLI и manifest.

## P2 — надежность, UX и качество

- [ ] Продолжать audit после ошибки отдельной платформы и отражать ошибку в структурированном результате, сводке и детальном отчете, не теряя результаты исправных платформ.
- [ ] Отключить интерполяцию `ConfigParser`, атомарно сохранять INI и показывать понятные ошибки autosave и открытия папок. Добавить регрессию для пути с `%` и сбоя atomic replace.
- [ ] Унифицировать безопасную запись отчетов: atomic replace, нейтрализация CSV-формул с префиксами `=`, `+`, `-`, `@`, стабильные имена с защитой от DOS reserved names и manifest для очистки только собственных stale-файлов. Покрыть `CON`, коллизии имен, смену режима и удаленные платформы.
- [ ] Зафиксировать допустимый XML round-trip либо сохранять comments и processing instructions; добавить post-write semantic validation и регрессии для comments, PI, namespaces и неожиданно больших XML.
- [ ] Добавить единый operation controller для CLI и GUI: immutable run-context, блокировку релевантных controls, прогресс и отмену read-only / pre-commit стадий, а также открытие фактической папки отчетов текущего запуска.
- [ ] Устранить утечку `Tooltip`; добавить keyboard focus, ограничение ширины и позиции подсказок, доступные локализованные названия кнопок выбора/открытия папок и адаптивный layout для RU/EN при 100–200% DPI.
- [ ] Оптимизировать после benchmark: получать один snapshot процессов на safety-фазу, обрабатывать XML последовательно без удержания всех деревьев, кэшировать одинаковые ROM-каталоги и использовать `scandir` без обхода junction. Добавить synthetic-тест большой коллекции и безопасную отмену.
- [ ] Разделить тесты по подсистемам, сделать GUI smoke обязательным, добавить `ruff`, постепенный `mypy`, Python 3.14 в CI и заменить deprecated `locale.getdefaultlocale` до Python 3.15.
- [ ] Усилить выпуск nightly EXE: зафиксировать build-зависимости и GitHub Actions по commit SHA, минимизировать permissions и публиковать SHA-256, SBOM и provenance; предусмотреть Authenticode-подпись.

## P3 — сопровождаемость и backlog

- [ ] После стабилизации operation controller декомпозировать `launchbox_tools/gui/app.py`: разделить каркас окна, панели операций, управление состоянием и форматирование результатов; унифицировать result/summary models и удалить дублирование CLI/GUI. Зафиксировать целевую структуру в `ARCHITECTURE.md`.
- [ ] Возвращать ошибку для неизвестного `--platform`, удалить неиспользуемый код и определить явную политику хранения, восстановления и очистки backup.
- [ ] Синхронизировать README с текущей навигацией, GUI path replacement, одной language-toggle кнопкой, фактическими обозначениями выбора/открытия папок, режимом отчетов и состояниями операций.

### Функциональный backlog

- [ ] Добавить восстановление основного файла из `AdditionalApplication` при его отсутствии.
- [ ] Добавить выгрузку избранного в структуру папок для портативки с выбором режима: только основной файл или все файлы.
- [ ] Реализовать поддержку многодисковых игр.
- [ ] Массовый аудит готовности игр к запуску. Проверять цепочку ROM → EmulatorId → executable эмулятора → associated/default platform → RetroArch core → явно зарегистрированные BIOS/dependencies. V1 только формирует сводку по играм и платформам без запуска программ и изменения конфигурации. Такой сценарий автоматизирует официальный ручной checklist LaunchBox.
- [ ] Аудит внутренней целостности базы. Находить повторяющиеся или пустые Game ID, orphaned AdditionalApplication и PlaylistGame, ссылки на отсутствующие Emulator ID, несогласованные платформы и дубли идентификаторов. V1 — строго read-only с отдельным отчётом по типам нарушений. Связи Game, Emulator и AdditionalApplication подтверждаются публичной моделью LaunchBox. 
- [ ] Поиск и перепривязка перемещённых или переименованных ROM по хешу. Сопоставлять missing_on_disk и not_in_database внутри одной платформы сначала по размеру, затем по SHA-256. Dry-run показывает кандидатов; apply разрешён только для уникального соответствия 1:1, неоднозначные совпадения остаются в отчёте. V1 не пытается считать одинаковыми ROM, перепакованные из ZIP в 7z. Это сохраняет metadata и media без удаления и повторного импорта.
- [ ] Просмотр, сравнение и выборочное восстановление backup. Проверять читаемость встроенных архивов LaunchBox и backup утилиты, сравнивать их с текущим Data по платформам, играм и Additional Apps. Восстановление V1 выполняется только целым выбранным XML-файлом, предварительно сохраняя текущую версию; element-level merge не поддерживается. LaunchBox уже создаёт Data backup, но не заменяет предлагаемый diff/validation workflow.
- [ ] Аудит переносимости сборки. Формировать manifest абсолютных, внешних и отсутствующих путей для Games, Additional Apps, эмуляторов, конфигурационных файлов и явно описанных dependencies. Для каждого ресурса указывать принадлежность корню LaunchBox, существование, размер и рекомендуемое действие. V1 только читает данные и ничего не копирует, поэтому не пересекается с экспортом избранного.
