# ADR-0001: Локальный MCP-сервер для управления коллекцией LaunchBox

- **Статус:** Proposed
- **Дата:** 2026-07-11

## Контекст

LaunchBox Utils развивается как набор детерминированных операций, которые
человек запускает через CLI или GUI. Большая коллекция ROM требует повторяющихся
аудитов, сопоставления данных из нескольких XML и файловых каталогов, объяснения
причин ошибок и подготовки массовых исправлений. Эти составные сценарии подходят
для ИИ-агента, если агент вызывает узкие типизированные операции, а не получает
произвольный доступ к XML, командной строке или файловой системе.

MCP-сервер сам не должен содержать LLM. Модель и диалог находятся в MCP-host, а
сервер остаётся локальным, детерминированным и независимым от поставщика модели.
Его задача — предоставить безопасный доменный интерфейс к общей логике
LaunchBox Utils.

Текущая архитектура уже отделяет `operations/*` от CLI и GUI, поэтому MCP может
стать третьим адаптером. Однако существующий mutation flow ещё не обеспечивает
многофайловую транзакцию, rollback, межпроцессную блокировку, неизменяемый
manifest и проверку исходного snapshot перед commit. Эти пробелы перечислены в
[`ROADMAP.md`](../../ROADMAP.md) как P0–P1.

## Заключение

Локальный MCP-сервер целесообразен как новая продуктовая поверхность LaunchBox
Utils. Первый релиз должен быть read-only и уметь формировать проверяемые планы,
но не применять их от имени агента. Agent-controlled apply допустим только после
реализации общего транзакционного mutation flow и проверки пользовательского
подтверждения.

Рыночный спрос на самостоятельный коммерческий продукт пока не подтверждён.
Поэтому направление следует проверить ограниченным MCP MVP, не отделяя его от
текущего репозитория и не превращая в обязательство roadmap до отдельного
решения.

## Решение

Развивать MCP как отдельный процесс и дистрибутив в том же репозитории:

```text
MCP host
    -> local stdio
LaunchBox Utils MCP adapter
    -> policy, schemas, jobs, plans, audit log
Shared application/domain core
    -> XML repository, audits, ROM index, transaction manager
```

- `launchbox_tools` остаётся единственным владельцем XML-модели, проверок,
  backup и безопасной записи.
- MCP не вызывает существующий CLI как subprocess и не редактирует XML
  самостоятельно.
- MCP получает отдельный entrypoint, конфигурационный профиль, журнал и
  PyInstaller artifact; официальный Python MCP SDK подключается как optional
  dependency.
- Первый transport — `stdio`. Сетевой transport не входит в MVP.
- LaunchBox root задаётся доверенной конфигурацией запуска. Аргументы модели не
  могут расширять разрешённые каталоги.
- Внешний Python-процесс может изменять XML только когда LaunchBox и Big Box
  закрыты. Работа с актуальным in-memory состоянием остаётся задачей будущего
  plugin bridge.

Таким образом, для пользователя MCP является отдельным компонентом LaunchBox
Utils, но не отдельным кодовым продуктом или репозиторием.

## Функциональные этапы

### 1. Read-only MVP

- состояние сервера и краткая сводка коллекции;
- список платформ, эмуляторов и плейлистов;
- поиск игр с фильтрами, стабильными ID, лимитами и пагинацией;
- сведения об игре и связанных Additional Applications;
- аудит отсутствующих и не зарегистрированных ROM;
- проверка целостности ID, ссылок и XML;
- preview дедупликации Additional Applications и замены путей;
- просмотр и сравнение backup;
- progress, status и cancellation для долгих аудитов.

Результаты возвращаются как ограниченные структурированные объекты. Сырые XML и
произвольные `file://` resources не публикуются. Для повторных запросов можно
использовать производный SQLite/FTS и hash-индекс; XML остаётся source of truth.
Векторная база не является требованием MVP.

### 2. Planning

- неизменяемые планы замены путей и консервативной дедупликации;
- hash-based поиск однозначно перемещённых ROM;
- portability manifest;
- сравнение установок и подготовка восстановления из backup.

План содержит `plan_id`, тип операции, нормализованные параметры, исходные
SHA-256, целевые XML, точное число изменений, bounded diff, предупреждения и
срок действия. На этом этапе агент не изменяет XML или ROM-файлы.

### 3. Guarded apply

Apply открывается только после завершения P0–P1 и общего operation controller.
Обязательный flow:

1. `plan` создаёт неизменяемый одноразовый план.
2. Пользователь подтверждает именно этот план через доверенный MCP-host или
   локальный UI. Поле `confirm: true` от модели подтверждением не считается.
3. `commit(plan_id)` повторно проверяет исходные hashes, разрешения, состояние
   LaunchBox, lock и доступное место.
4. Выполняется `backup all -> stage all -> validate all -> commit -> verify`.
5. Частичный сбой приводит к rollback и фиксируется в operation manifest.

Первые допустимые мутации: замена путей, канонически безопасная дедупликация,
relink по уникальному совпадению 1:1 и восстановление целого XML из проверенного
backup.

## Ограничения безопасности

- Режим по умолчанию — read-only; write-права включаются только доверенной
  конфигурацией запуска.
- Не предоставляются generic `read_file`, `write_file`, XPath/XML patch,
  shell-команды и произвольный доступ к файловой системе.
- Удаление ROM и постоянное удаление backup не входят в начальные версии.
- Все пути проходят canonical containment checks; UNC, traversal и выход через
  symlink/junction запрещены по умолчанию.
- Между всеми MCP, CLI и GUI apply действует единый межпроцессный single-writer
  lock.
- Данные из названий, Notes и других XML-полей считаются недоверенными и никогда
  не используются как инструкции или динамические описания tools.
- Ответы имеют пагинацию, ограничения размера, timeout и cancellation.
- «Локальный MCP» не означает, что данные останутся на компьютере: MCP-host может
  передать результаты облачной модели, поэтому сервер возвращает только
  необходимые поля.

## Будущий C# Plugin API bridge

Официальный LaunchBox Plugin API предоставляет `IDataManager` для чтения и
записи данных, а `BackgroundReloadSave(Action)` перечитывает актуальные данные
перед небольшим изменением и сохранением. В перспективе тонкий C# plugin может
предоставить локальный named-pipe bridge:

```text
LaunchBox Utils MCP -> authenticated named pipe -> LaunchBox plugin -> IDataManager
```

MCP transport, policy и журналы остаются во внешнем процессе. Plugin нужен для
доступа к актуальному in-memory состоянию и безопасных изменений при работающем
LaunchBox, но не является условием read-only MVP.

## Рассмотренные альтернативы

### Отдельный репозиторий и самостоятельное ядро

Отклонено на начальном этапе: приведёт к дублированию XML parsing, path rules,
safety checks и тестов. Разделение возможно после стабилизации публичного
application API или появления независимой команды и цикла релизов.

### MCP внутри Tkinter GUI

Отклонено: MCP имеет другой жизненный цикл, протокол stdout и профиль
безопасности. Ошибка или завершение GUI не должны прерывать серверный запрос или
commit.

### MCP полностью внутри C# plugin

Отклонено как стартовая архитектура: сервер будет доступен только при запущенном
LaunchBox, а MCP-зависимости и поверхность атаки окажутся внутри его процесса.
Plugin bridge остаётся дополнительным adapter.

### Прямой доступ агента к XML, SQLite или shell

Отклонено: универсальные операции имеют чрезмерный blast radius, плохо
проверяются и позволяют модели обходить доменные инварианты.

## Последствия

- Потребуется выделить общий application controller и стабильные DTO для CLI,
  GUI и MCP.
- Read-only MVP можно разрабатывать до завершения mutation P0; apply — нельзя.
- Появится optional runtime dependency и отдельный release artifact.
- Для больших коллекций потребуется измеряемый локальный индекс, но не отдельное
  облачное хранилище или встроенная LLM.
- Необходимо тестировать MCP schemas, pagination, несколько одновременно
  запущенных stdio-серверов и операции на больших фиктивных коллекциях.

## Условия принятия и переноса в roadmap

Статус можно изменить на `Accepted`, когда:

- подтверждены минимум несколько реальных составных пользовательских сценариев,
  которые заметно удобнее CLI/GUI;
- определены границы read-only MVP и стабильные tool schemas;
- проверена работа хотя бы с несколькими распространёнными MCP-host;
- benchmark подтверждает приемлемую работу на большой коллекции;
- согласованы packaging, optional dependency и security policy.

После принятия в roadmap добавляется только read-only MVP. Planning и guarded
apply оформляются отдельными задачами после появления соответствующих
транзакционных гарантий.
